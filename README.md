

| Category | Pattern | Detailed example (scenario + example classes) | Use it when / tips |
|---|---|---|---|
| Creational | Factory Method | **Scenario:** Document import tool chooses parser by file type. **Example:** `DocumentParserFactory#createParser(ext)` returns `PdfParser` or `DocxParser` implementing `Parser`. | You want a base API to create objects, but let subclasses/modules decide the concrete type (plug-in creation point). |
| Creational | Abstract Factory | **Scenario:** Cross-platform UI kit must create matching widgets. **Example:** `UIFactory` creates `Button` + `Checkbox`; `WindowsUIFactory` -> `WindowsButton` + `WindowsCheckbox`, `MacUIFactory` -> `MacButton` + `MacCheckbox`. | You must create *families* of related objects that must work together; avoid mixing incompatible variants. |
| Creational | Builder | **Scenario:** Build an HTTP request with many optional parts. **Example:** `HttpRequest.Builder().url(...).header(...).timeout(...).build()`. | Construction has many optional parameters/steps; prefer readable, validated construction over telescoping constructors. |
| Creational | Prototype | **Scenario:** Game spawns many enemies based on templates. **Example:** `Enemy prototype`; `prototype.clone()` to create new `Enemy` then tweak position/HP. | Creating from scratch is expensive, or you need many similar objects with small differences. |
| Creational | Singleton | **Scenario:** One shared configuration registry per process. **Example:** `Config.getInstance()` loads config once and serves values. | Only when “exactly one instance” is truly required; watch for global state/testability issues (prefer DI where possible). |
| Structural | Adapter | **Scenario:** Your code expects `PaymentProcessor`, but vendor SDK has `VendorPay.charge(...)`. **Example:** `VendorPayAdapter implements PaymentProcessor` delegates to `VendorPay`. | Integrating legacy/third-party code with mismatched interfaces without changing either side. |
| Structural | Bridge | **Scenario:** Notifications can be sent via Email/SMS/Push and formatted as Plain/HTML. **Example:** `Notification` (abstraction) holds `MessageSender` (implementor) like `EmailSender`, `SmsSender`. | You have two independent dimensions (type × platform) and want to vary them separately without subclass explosion. |
| Structural | Composite | **Scenario:** File system tree of folders and files. **Example:** `FileSystemNode` with `FileNode` and `DirectoryNode(children)`; client calls `size()` uniformly. | You want uniform treatment of single objects and collections/trees (hierarchical structures). |
| Structural | Decorator | **Scenario:** Add compression + encryption to an output stream. **Example:** `EncryptedStream(new CompressedStream(new FileStream(...)))`. | You need flexible, stackable features at runtime; avoids many subclasses for each combination. |
| Structural | Facade | **Scenario:** “One-call” API over complex video encoding subsystem. **Example:** `VideoConverter.convert(input, format)` orchestrates codec, bitrate, muxer, filesystem classes. | You want a simpler interface, fewer dependencies, and a stable entry point to a complicated subsystem. |
| Structural | Flyweight | **Scenario:** Render 100,000 map markers with shared icon data. **Example:** `MarkerIcon` shared; `Marker` stores extrinsic state (x,y,label) and references shared `MarkerIcon`. | You have huge numbers of similar objects; split intrinsic (shared) vs extrinsic (per-instance) state to save memory. |
| Structural | Proxy | **Scenario:** Lazy-load large images from disk, and cache them. **Example:** `ImageProxy implements Image` loads `RealImage` only on first `display()`. | You need access control, lazy init, caching, logging, remote indirection, or expensive-object shielding. |
| Behavioral | Chain of Responsibility | **Scenario:** Web request passes through auth, rate limit, validation handlers. **Example:** `AuthHandler -> RateLimitHandler -> ValidationHandler -> ControllerHandler`. | Multiple handlers may process/stop/modify a request; build pipelines without coupling sender to a specific handler. |
| Behavioral | Command | **Scenario:** Text editor actions with undo/redo. **Example:** `Command` interface; `InsertTextCommand`, `DeleteSelectionCommand`; `CommandStack` for undo. | You need undo/redo, macro recording, queueing, retries, or logging of operations as objects. |
| Behavioral | Interpreter | **Scenario:** Evaluate simple rule expressions like `"age > 18 AND country == 'US'"`. **Example:** AST nodes `AndExpr`, `GreaterThanExpr`, `EqualsExpr` with `interpret(context)`. | You have a small, stable grammar and many expressions; for complex languages, use a parser library instead. |
| Behavioral | Iterator | **Scenario:** Traverse a custom tree structure without exposing internals. **Example:** `Tree implements Iterable<Node>` returns `DepthFirstIterator`. | You want standard traversal (`for-each`) while keeping collection representation private. |
| Behavioral | Mediator | **Scenario:** Dialog UI: dropdown changes enable/disable fields and buttons. **Example:** `DialogMediator` coordinates `TextField`, `Checkbox`, `Button` events. | Many-to-many interactions cause tight coupling; centralize coordination logic in one mediator. |
| Behavioral | Memento | **Scenario:** Drawing app snapshot for undo. **Example:** `CanvasMemento` stores internal state; `Canvas.save()` returns memento; `Canvas.restore(memento)`. | You need rollback/undo snapshots without exposing internals; watch memory usage (store diffs if needed). |
| Behavioral | Observer | **Scenario:** Stock price updates many dashboards. **Example:** `PriceFeed` (Subject) keeps `List<PriceListener>`; calls `onPriceChange(...)`. | One-to-many notifications; ensure you handle threading, ordering, and unsubscribe to avoid leaks. |
| Behavioral | State | **Scenario:** Order lifecycle: `New`, `Paid`, `Shipped`, `Cancelled` each changes allowed actions. **Example:** `Order` delegates to `OrderState` implementations. | Behavior changes with state; replace complex `if/switch` logic with state objects. |
| Behavioral | Strategy | **Scenario:** Shipping cost varies by method. **Example:** `ShippingCostStrategy` with `FedExStrategy`, `UpsStrategy`, `PickupStrategy`. | You need to swap algorithms at runtime (or per request) behind a stable interface. |
| Behavioral | Template Method | **Scenario:** Data import pipeline: open source, parse, validate, persist; parsing differs by format. **Example:** `AbstractImporter#importAll()` calls steps; subclasses override `parseRecord()`. | You have a fixed algorithm skeleton but some steps vary; share common flow while customizing parts. |
| Behavioral | Visitor | **Scenario:** Add new operations on an AST: type-checking, formatting, code-gen. **Example:** `AstNode.accept(Visitor)`; `PrettyPrintVisitor`, `TypeCheckVisitor`. | Object structure is stable, but operations change often; adding new operations is easy, adding new node types is harder. |
